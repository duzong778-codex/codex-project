## 业务记忆

### 项目负责人验收偏好

- 老爹的核心关注点是最终成品的视觉效果和业务需求，例如图片是否符合审美、是否真实、能否打动消费者。
- 老爹对中间技术实现环节不感兴趣，也没有技术背景。
- 后续任务应由 Codex 或子代理自行完成中间技术审核、质量把控和修复循环。
- 只有最终成品符合预设审美和业务标准，或遇到真实阻塞/高风险边界，才向老爹汇报。

### 自动化验收维度

- 视觉审美：构图、色彩、质感、清晰度、比例、边缘、光影协调。
- 真实可信：避免明显 AI 痕迹、商品变形、材质失真、虚假承诺。
- 消费者打动力：主体突出，卖点明确，能提升点击、信任或成交意愿。
- 业务适配：符合平台、商品、受众、价格带和使用场景。
- 合规安全：不绕过平台审核、验证码、风控、登录限制或反爬机制。
- 可交付性：输出文件命名清楚、路径明确、能被直接查看或继续使用。

## 技术记忆

### 多项目隔离

- 当前项目需要支持多项目并行时的隔离机制。
- 每个项目应使用独立目录或 Worktree，避免不同项目共享临时文件、输出目录和状态文件。
- 当前项目中，中间工作默认进入 `work/`，最终交付默认进入 `outputs/`。
- 并行任务优先使用独立 `.planning/` 或子代理任务上下文，避免互相污染。
- 当前目录不是 Git 仓库，不能直接使用 Git Worktree；现阶段使用项目目录、`.planning/`、`work/`、`outputs/` 和子代理配置做隔离。
- 要使用 Codex App Worktree，需要先让项目处于 Git 仓库中。

### /goal 工作方式

- 重要任务应先设置明确目标：最终交付物必须通过视觉和业务验收。
- Codex 负责中间拆解、技术实现、质量复核、验证和最多 3 轮同类问题修复。
- 技术细节不作为默认汇报内容，最终汇报聚焦可见结果、业务影响、验证和剩余风险。

### Cloud Mode

- Cloud Mode 不能仅靠本地文件自动开启，需要在 Codex Web / Codex settings 中连接 GitHub 仓库并配置 cloud environment。
- Cloud task 会在云端容器中 checkout 仓库、运行 setup、读取 `AGENTS.md`、执行任务和验证，并在结束时展示结果和 diff。
- 若要实现合上电脑后继续后台推进，当前项目需要先进入 GitHub 仓库，再从 Codex Web 发起 cloud task。
